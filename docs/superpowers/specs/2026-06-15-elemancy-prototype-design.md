# Elemancy — 원소 연금술 조합 디펜스 (프로토타입 설계)

- 버전: 0.1.0 (설계)
- 작성일: 2026-06-15
- 단계: **1단계 프로토타입** (재미 검증용, 출시용 아님)

---

## 1. 목표와 범위

### 1.1 이 단계에서 증명할 것

> "이 조합 디펜스가 실제로 손이 가고 재밌는가."

재미가 확인되기 전에는 2단계(Godot 출시 빌드)로 넘어가지 않는다. 인디 게임 실패의 가장 흔한 원인이 검증 없는 본 제작이므로, 1단계의 유일한 성공 기준은 **플레이 가능한 프로토타입으로 친구·커뮤니티 반응을 확인**하는 것이다.

### 1.2 장르

원랜디(원소 랜덤 디펜스, 워크3 유즈맵 계열) 스타일의 **싱글플레이 타워 디펜스**.

- 주기적으로 **랜덤 원소**가 지급된다 (tier 1).
- 받은 원소를 합성 트리대로 **조합**해 상위 타워로 키운다.
- 네모난 순환 경로를 **계속 도는 웨이브 적**을 타워로 처치한다.
- 적은 죽을 때까지 무한 순환하며, **보드 위 생존 몬스터가 100마리를 넘으면 게임오버**. 40웨이브 클리어 = 승리.

### 1.3 전략의 두 축

원소 상성(가위바위보)은 **쓰지 않는다**. 대신:

1. **합성 tier 깊이** — 적 구간마다 권장 대응 단계(`reqTier`)가 있고, 보스는 해당 단계 유닛 없이는 사실상 못 막는다.
2. **적 역할 대응** — 한 웨이브에 역할(swarm·tank·fast·special)이 섞여 나와, 단일 조합으로는 못 막고 공격 타입을 다양하게 갖춰야 한다.

### 1.4 명시적 비범위 (YAGNI)

아트·사운드 완성도, 세이브/옵션, 멀티플레이, 실시간 준비 압박, 모바일, 수익화는 **전부 1단계 제외**. 렌더는 도형·색 플레이스홀더로 충분하다.

## 2. 기술 스택

- **순수 HTML5 + Canvas + 바닐라 JS** (의존성 0, 브라우저에서 정적 파일로 바로 실행).
- 기존 `alchemy.js` / `wavesystem.js`가 Node·브라우저 양쪽 호환이라 그대로 코어로 재사용.
- 2단계(Godot) 이전 시 `recipes.json` / `enemies.json`은 그대로 옮겨진다.

## 3. 모듈 구조

데이터(무엇을 만드나)와 로직(어떻게 처리하나) 분리 철학을 그대로 이어간다.

| 레이어 | 파일 | 상태 | 책임 |
|---|---|---|---|
| 데이터 | `data/recipes.json` | 재사용 + 필드 약간 추가 | 원소·합성 트리 |
| 데이터 | `data/enemies.json` | 재사용 | 적·웨이브 구성 |
| 합성 로직 | `src/alchemy.js` | 그대로 | 조회/검증/탐색/원가 |
| 웨이브 로직 | `src/wavesystem.js` | 그대로 | 웨이브별 적 구성 |
| **전투 엔진** | `src/combat.js` | 신규 | 타겟팅·타격·데미지·슬로우·버프·광역 |
| **타워 능력치** | `src/stats.js` | 신규 | 계열·공격타입·데미지 밴드·공속·사거리 산출 |
| **게임 상태** | `src/game.js` | 신규 | 인벤토리·골드·라이프·웨이브 진행·배치·경제 |
| **렌더/입력** | `src/render.js`, `index.html` | 신규 | Canvas 그리기, 클릭/조합 UI |
| 진입점 | `src/main.js` | 신규 | 데이터 로드 + `requestAnimationFrame` 루프 |

각 모듈의 경계:
- `stats.js`: 유닛 id를 받아 `{family, atkType, damage, atkSpeed, range}` 반환. 순수 함수, 외부 의존은 `alchemy.js`(baseCost)뿐.
- `combat.js`: 타워 목록 + 적 목록 + dt를 받아 한 틱의 전투를 진행. 렌더·입력을 모른다.
- `game.js`: 상태의 단일 소스. 조합/구매/배치/웨이브 전이를 메서드로 노출.
- `render.js`: `game` 상태를 읽어 그리기만. 상태를 직접 수정하지 않고 콜백으로 의도를 전달.

## 4. 핵심 게임 루프

```
[준비 단계 — 무한, "다음 웨이브" 누를 때까지]
  · 원소 클릭 → 조합 패널(이 원소로 만들 수 있는 유닛 목록)
  · 재료 다 모이면 [조합] 버튼 활성 → 클릭 시 합성
  · 골드로 원하는 tier1 원소 직접 구매
  · 타워를 빈 슬롯에 배치
        │
        ▼  "다음 웨이브"
[전투 단계]
  · 적 경로 진입 → 타워 자동 공격(사거리 내)
  · 처치 → 골드 / 누수 → 라이프 −
  · 전멸 시 웨이브 클리어
        │
        ▼
[보상] 랜덤 tier1 원소 N개 지급 → 다시 준비 단계
```

- 매 라운드 약 20마리 스폰. 적은 죽을 때까지 순환 경로를 무한히 돈다(누수·라이프 없음).
- **게임오버 = 보드 위 생존 몬스터 수 > 100.** 화면에 `현재/100` 위기 게이지를 크게 표시.
- 보스 웨이브(10/20/30/40)는 **해당 구간 일반 적 + 보스 동반 출현** (보스 단독 아님). 예: w10 = 1,000 HP 일반 몬스터들 + 3,000 HP 보스가 함께 등장.
- **보스 캡 가중치**: 보스 1마리는 누적 캡에서 **8마리로 카운트**. 안 잡으면 캡을 빠르게 압박해 tier 게이트 긴장을 유지.
- 40웨이브 클리어 = 승리.

## 5. 전투 수치

핵심 원리: **데미지 ×10/티어, 적 HP ×10/구간**으로 맞물려, 어느 구간이든 그 단계 유닛으로 **약 10대**면 잡히게 한다.

### 5.1 적 HP

w1 = 100에서 시작, **라운드마다 `100 × 10^(구간-1)` 누적**.

| 구간 (reqTier) | 라운드당 증가 | 구간 끝 HP | 그 단계 유닛 데미지 | 처치 타수 |
|---|---|---|---|---|
| seg1 (t2) w1–10 | +100 | w10 = 1,000 | t2: 100–200 | ~5–10대 |
| seg2 (t3) w11–20 | +1,000 | w20 = 11,000 | t3: 1,000–2,000 | ~6–11대 |
| seg3 (t4) w21–30 | +10,000 | w30 = 111,000 | t4: 10,000–20,000 | ~6–11대 |
| seg4 (t5) w31–40 | +100,000 | w40 = 1,111,000 | t5: 100,000–200,000 | ~6–11대 |

구간 시작은 여유, 구간 끝(보스 직전)이 가장 빡빡한 곡선. 구간 시작은 의도된 숨고르기 구간이다.

**역할 HP 배수** (위 base HP에 곱함):

| 역할 | 배수 |
|---|---|
| swarm | ×0.3 |
| fast | ×0.5 |
| special | ×1.2 |
| tank | ×2.5 |
| boss | **구간 끝 라운드 base HP × 3** (아래 표) |

보스는 같은 웨이브의 일반 적들과 **함께** 등장한다(보스 단독 아님). 따라서 `wavesystem.js`의 `composeWave()`를 수정 — 보스 웨이브는 일반 구성 결과에 보스 1마리를 추가한다.

**보스 HP** (역할 배수 대신 고정식 = 구간 끝 base × 3):

| 보스 웨이브 | 구간 끝 base HP | 보스 HP |
|---|---|---|
| w10 | 1,000 | 3,000 |
| w20 | 11,000 | 33,000 |
| w30 | 111,000 | 333,000 |
| w40 | 1,111,000 | 3,333,000 |

> `enemies.json`의 기존 `hp` 리터럴 값은 참고용이며, 실제 HP는 위 공식으로 계산한다(웨이브 번호 + 역할). `count`/`role`/`segment`/`reqTier`는 그대로 사용.

### 5.2 원소 계열별 타워 능력치

타워는 생성 시 데미지 밴드 내에서 **랜덤 값**을 굴려 개체차를 둔다.

| 계열 | 공격 타입 | tier1 데미지 밴드 | 공속 | 비고 |
|---|---|---|---|---|
| 번개 | 광역 | 15–20 | 1.5 | 빠른 광역 |
| 불 | 광역 | 10–15 | 1.0 | 광역 |
| 금속 | 단일 고화력 | 15–20 | 1.0 | 최고 단일딜 (탱커 처리) |
| 바람 | 단일 고화력 | 10–15 | 1.5 | 빠른 단일 |
| 얼음 | 둔화 | 10–15 | 1.0 | 명중 시 이동속도 −40% (2초) |
| 물 | 둔화 | 10–15 | 1.0 | 명중 시 이동속도 −40% (2초) |
| 나무 | 버프 | — (공격 안 함) | — | 주변 타워 공속 1.5–2.0, 공격력 +25% |
| 흙 | 버프 | — (공격 안 함) | — | 주변 타워 공속 1.5–2.0, 공격력 +25% |

- **데미지 밴드는 티어마다 ×10** (t2 = ×10, t3 = ×100, t4 = ×1,000, t5 = ×10,000). 금속·번개 계열만 상위 밴드(15–20×).
- **공속**: 기본 1.0, 번개·바람 계열만 1.5.
- **사거리** (tier별): t1 2.0 / t2 2.2 / t3 2.5 / t4 2.8 / t5 3.2 (격자 칸 단위, 튜닝값).

**공격 타입 효과 정리**

| 타입 | 효과 | 주 대응 |
|---|---|---|
| 광역 | 착탄 반경 내 전체 타격 | swarm |
| 단일 고화력 | 단일 대상에 밴드 데미지 | tank |
| 둔화 | 명중 시 −40% 2초 + 데미지 | fast |
| 버프 | 공격 안 함, 주변 타워 공격력 +25% / 공속 1.5–2.0 | 아군 강화 |

### 5.3 상위 유닛 계열 결정

상위 유닛(증기·폭풍·은하 등)은 여러 원소로 구성되므로:

- **기본: 지배 원소 자동 산출** — `alchemy.js`의 `baseCost(id)`로 1단계 원소 구성을 펼친 뒤, **가장 많은 계열**의 (타입·밴드·공속)을 따른다.
- **동률: `inputs[0]`** (첫 재료)의 계열로 결정.
- **예외: 명시 오버라이드** — 특정 유닛만 `recipes.json`에 `atkType`(또는 `family`) 필드를 직접 넣으면 그 값이 우선.

예) 증기(물+불) → 물1·불1 동률 → `inputs[0]`=물 → **둔화**. 폭풍(번개·구름·안개) → 전기 계열 다수 → **광역**.

## 6. 맵 & 배치

원랜디 정석 레이아웃: **네모난 순환 경로(사각형 트랙)** 를 적이 계속 돈다.

```
┌─────────────────────────────┐   ← 바깥 경로(적이 시계방향으로 순환)
│  ┌───────────────────────┐  │
│  │                       │  │
│  │   중앙 빌드 구역       │  │   ← 타워는 사각형 안쪽 격자에 배치
│  │   (타워 격자)         │  │
│  │                       │  │
│  └───────────────────────┘  │
└─────────────────────────────┘
     ▲ 적 스폰                   적이 경로를 돌며 사거리 안 타워에 맞음
```

- **순환 경로** — 적은 사각형 둘레 경로를 **죽을 때까지 무한히** 돈다. 타워는 사각형 **안쪽 빌드 격자**(및 필요 시 바깥 가장자리)에 놓아, 사거리로 경로를 때린다.
- **게임오버 조건** — 누수·라이프 개념 없음. 보드 위 **생존 몬스터 수가 100을 초과하면 패배**. 매 라운드 ~20마리가 추가되므로, 스폰 속도보다 빠르게 처치하지 못하면 보드가 차올라 진다. (보스는 8마리로 카운트 — §5.1)
- **소유 = 보드 위** — 받은/조합한 유닛은 빌드 격자 슬롯에 놓인다. 별도 인벤토리 창 없이 보드에서 바로 관리.
- **조합 동작** — 유닛 클릭 → 조합 패널 → 재료가 다 있으면 [조합] → 재료 타워들이 사라지고 결과 타워가 빈자리에 생성.
- 재료가 보드 곳곳에 흩어져 있어도 소유 기준으로 합성 가능(보유 개수 기반, `alchemy.js`의 인벤토리 모델과 일치).

## 7. 경제 & 흐름

### 7.1 골드

적 처치 시 획득(역할·tier별 차등). 용도는 세 가지:

**(1) 공격력 업그레이드 (티어별)**
- 각 티어(1~5)마다 독립된 업그레이드 레벨을 가진다.
- **레벨당 해당 티어 타워 데미지 +1%**, **최대 1000레벨(= +1000%, 즉 ×11)**.
- **초기 비용을 저렴하게** 설정해 업그레이드를 유도하고, 비용은 **티어·레벨에 비례** 증가(상위 티어일수록 비쌈).
- 시작 공식(튜닝값): `다음레벨 비용 = ceil(티어 × 20 × 1.01^현재레벨)` — tier1 첫 레벨 ≈ 20골드로 저렴하게 시작.

**(2) 도박**
- 베팅액 100 / 500 / 1000 중 선택.
- 결과 = **순손익이 −베팅액 ~ +베팅액 사이 균등 랜덤** (기대값 0, 순수 운치기).
  - 예: 1000 베팅 → −1000 ~ +1000 사이 임의 정수만큼 골드 증감.

**(3) 원소 가챠 (티어별 뽑기)**

| 베팅 금액 | 대상 | 성공 확률 | 성공 시 |
|---|---|---|---|
| 1,000 | tier1 | 50% | 해당 티어 유닛 중 랜덤 1개 획득 |
| 5,000 | tier2 | 40% | 해당 티어 유닛 중 랜덤 1개 획득 |
| 10,000 | tier3 | 30% | 해당 티어 유닛 중 랜덤 1개 획득 |

- 실패 시 골드만 소모하고 꽝.

### 7.2 무료 / 보상 지급

| 시점 | 지급 |
|---|---|
| 게임 시작 | 랜덤 tier1 원소 **5개** |
| 웨이브 클리어 | tier1 원소 **2개** |
| 보스 처치 | **원소 선택권**(원하는 tier1 원소를 직접 골라 획득), 보스마다 **×3** — w10: 1개 / w20: 3개 / w30: 9개 / w40: 27개 |

### 7.3 흐름 / 게임오버

| 항목 | 규칙 |
|---|---|
| 스폰 | 매 라운드 ~20마리, 죽을 때까지 무한 순환 |
| 게임오버 | 보드 위 생존 몬스터 > 100 (보스는 8 카운트) |
| 준비 단계 | "다음 웨이브" 누를 때까지 **무한** (조합·구매·배치 시간 보장) |

실시간 압박(준비 시간 제한)은 1단계 제외 — 재미 검증 후 도입 여부 결정.

## 8. 데이터 추가 사항

- `recipes.json` — 1단계 원소 8종의 `family`는 코드 상수 맵으로 둔다(데이터 변경 최소화). 상위 레시피는 필요 시에만 `atkType`/`family` 오버라이드 필드 추가.
- `enemies.json` — 변경 없음. `hp` 리터럴은 참고용, 실제 HP는 §5.1 공식.

원소 → 계열 매핑 (코드 상수):

## 9. 테스트 전략

프로토타입의 진짜 검증은 **수동 플레이테스트**다. 다만 수치 회귀를 막기 위해 순수 로직만 가볍게 단위 테스트:

- `stats.js`: 계열 산출(동률·오버라이드 포함), 티어별 데미지 밴드, 공속/사거리.
- 적 HP 공식: 각 구간 끝 값, 역할 배수, 보스 ×3.
- `combat.js`: 단일/광역/둔화/버프 데미지 계산.
- `game.js`: 조합 시 재료 소모·결과 생성, 보드 몬스터 카운트(보스 8 가중치) 및 100 초과 게임오버 판정.
- 경제(`game.js`): 티어별 업그레이드 비용·데미지 배수(레벨당 +1%, 캡 1000), 도박 순손익 범위(−X~+X), 가챠 확률·소모, 보스 원소 선택권 수량(1·3·9·27).

## 10. 열린 튜닝 항목 (구현 후 플레이테스트로 조정)

- 데미지 밴드 절대값, 사거리, 공속 배수
- 골드 획득량(역할·tier 차등)
- 업그레이드 비용 공식(초기 저렴, `티어×20×1.01^레벨`), 가챠 확률·금액, 도박 베팅 단위
- 시작/웨이브/보스 지급량 (5개 / 2개 / 1·3·9·27)
- 역할 HP 배수, 보스 배수
- 라운드당 스폰 수(~20), 게임오버 캡(100), 보스 캡 가중치(8)
- 광역 반경, 둔화 수치·지속시간, 버프 반경·수치
